# ADR-004: Trait-Based Dependency Injection at Crate Boundaries

**Status:** Accepted
**Date:** 2026-03-13
**Deciders:** Platform Architecture Team
**Relates to:** ADR-002, all FRs

---

## Context

Protein-Engine's 14 crates form a deep dependency chain (pe-api -> pe-swarm -> pe-neural -> pe-vector -> pe-core). The London School TDD approach (ADR-002) requires that every downstream dependency can be replaced with a test double. Rust does not have a runtime DI container like Java's Spring or .NET's built-in DI. We need a compile-time mechanism to swap real and mock implementations.

## Decision

**Every inter-crate dependency is expressed as a Rust trait defined in the upstream (consuming) crate or in a shared trait crate.** Concrete implementations live in the downstream crate. Test doubles are generated by `mockall` from the trait definition.

Dependency injection is achieved through **generic type parameters bounded by traits** or **trait objects (`Box<dyn Trait>`)**, chosen per-boundary based on performance requirements:

- **Generics** (zero-cost, monomorphized): Used for hot paths — `EmbeddingModel`, `FitnessPredictor`, `VectorStore`
- **Trait objects** (dynamic dispatch): Used for collections of heterogeneous backends — `Vec<Box<dyn QuantumBackend>>`, agent role dispatch in pe-swarm

## Injection Patterns

### Generic (hot path)
```rust
pub struct EnsemblePredictor<T: SubModelScorer, L: SubModelScorer, N: SubModelScorer> {
    transformer: T,
    lstm: L,
    nbeats: N,
}
```

### Trait object (heterogeneous collection)
```rust
pub struct QuantumRouter {
    backends: Vec<Box<dyn QuantumBackend>>,
}

pub struct SwarmOrchestrator {
    agents: HashMap<AgentRole, Box<dyn SwarmAgent>>,
}
```

### Constructor injection (typical wiring)
```rust
impl ProteinEngineApi {
    pub fn new(
        scorer: Box<dyn FitnessPredictor>,
        store: Box<dyn VectorStore>,
        ledger: Box<dyn LedgerWriter>,
        quantum: Box<dyn QuantumBackend>,
    ) -> Self { ... }
}
```

## Consequences

### Positive
- Every inter-crate seam is testable in isolation via `#[automock]`
- Generics on hot paths have zero runtime overhead
- Trait objects for heterogeneous collections (quantum backends, agents) are natural Rust idiom
- New backends (e.g., a new quantum provider) plug in by implementing the trait

### Negative
- Generic type parameters propagate upward through the stack, increasing signature verbosity
- Trait objects require `Send + Sync` bounds for use in async contexts
- `mockall` does not support all trait patterns (e.g., GATs require workarounds)

### Rules
1. A trait is created **only** when it crosses a crate boundary or represents a genuinely swappable backend
2. Within a single crate, prefer concrete types — do not introduce traits for internal helpers
3. All traits must be `Send + Sync` for compatibility with both tokio (native) and single-threaded (WASM) runtimes
4. Async traits use `#[async_trait]` until Rust stabilizes native async trait support
