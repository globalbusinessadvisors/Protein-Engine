# ADR-002: London School TDD with mockall

**Status:** Accepted
**Date:** 2026-03-13
**Deciders:** Platform Architecture Team
**Relates to:** All FRs (testing strategy)

---

## Context

Protein-Engine is a 14-crate Rust workspace with deep dependency chains (pe-api -> pe-swarm -> pe-neural -> pe-vector -> pe-core). Testing each crate requires a strategy for isolating it from its downstream dependencies, which may involve hardware backends (quantum computers), external services (pyChemiQ sidecar), or expensive computations (ESM-2 inference).

Two TDD schools exist:
- **Detroit School (Classical)**: Use real collaborators wherever possible; test through the full stack; mocks only for truly external/slow dependencies
- **London School (Mockist)**: Mock all collaborators at trait boundaries; test only the unit's own logic; replace mocks with real implementations layer by layer

## Decision

**We adopt the London School (Outside-In) TDD approach using the `mockall` crate for test double generation.** Every crate boundary is defined by a trait. When testing crate X, all traits representing downstream crates Y and Z are replaced with mock implementations generated by `mockall`.

Development proceeds outside-in: acceptance tests at the API/CLI/WASM boundary drive the design inward. Inner crate interfaces are discovered by asking "what does this layer need from the layer below?" and expressing that as a trait.

## Rationale

- **Fast feedback loops**: Unit tests never touch real quantum backends, vector databases, or neural models â€” they complete in milliseconds
- **Design pressure**: Mandating traits at every boundary forces clean, narrow interfaces between crates; fat interfaces become immediately painful to mock
- **Incremental integration**: Each layer is proven in isolation before wiring to real collaborators; integration bugs are localized to the boundary where mocks are swapped out
- **Parallel development**: Teams can implement pe-quantum and pe-neural simultaneously because both depend on pe-core traits, not on each other's concrete types
- **Behavior verification**: London School naturally verifies that collaborators are called with correct arguments and in correct order (e.g., "verify mock_scorer.predict called 10 times")

## Mock Library: `mockall`

```toml
[workspace.dependencies]
mockall = { version = "0.12" }

# Per-crate:
[dev-dependencies]
mockall = { workspace = true }
```

`mockall` is chosen over alternatives because:
- Auto-generates `MockFoo` from `#[automock] trait Foo`
- Supports `#[async_trait]` for async methods (pe-quantum, pe-stream, pe-api)
- Supports expectation chaining: `.times(n)`, `.with(eq(...))`, `.returning(|_| ...)`
- Mature (0.12+), widely used in the Rust ecosystem

## Consequences

### Positive
- Test suite runs in < 5 seconds for the entire workspace (no I/O, no network)
- Every crate's public API is precisely defined by its trait surface
- Refactoring an inner crate cannot break outer crate tests (mocks absorb the change)
- New developers understand crate boundaries by reading trait definitions

### Negative
- Mocks can drift from real implementations â€” an integration test layer is required to catch this
- More boilerplate: every collaborator needs a trait even if there's only one implementation
- `mockall` generates code that can be opaque in compiler error messages
- Over-mocking risk: tests may verify implementation details rather than behavior

### Mitigations
- **Integration test layer**: Each phase includes integration tests that wire real crates together
- **Trait hygiene rule**: A trait is justified only if it crosses a crate boundary or represents a genuinely swappable backend
- **Behavior focus**: Test names describe outcomes ("search_nearest returns k closest"), not implementations ("calls inner.query once")

## Trait Boundary Map

| Consuming Crate | Trait | Mock |
|-----------------|-------|------|
| pe-vector | `EmbeddingModel` | `MockEmbedder` |
| pe-neural | `FitnessPredictor`, `TransformerScorer`, `LstmScorer`, `NBeatsScorer` | `MockScorer` variants |
| pe-swarm | `EvolutionEngine`, agent role traits | `MockEvolver`, `MockAgent` per role |
| pe-quantum | `QuantumBackend` | `MockBackend` |
| pe-ledger | `CryptoSigner`, `LedgerWriter` | `MockSigner`, `MockLedger` |
| pe-rvf | segment producer traits | `MockSegmentProducer` |
| pe-api | all domain traits | Full mock stack |
| pe-wasm | all domain traits | Stub stack |

## Outside-In Workflow

```
1. Write failing acceptance test at crate boundary
2. Identify collaborator traits needed
3. Generate mocks with #[automock]
4. Write unit tests using mocks
5. Make tests green with minimal implementation
6. Move inward: implement next downstream crate
7. Swap mock for real implementation
8. Re-run acceptance test against real stack
```
